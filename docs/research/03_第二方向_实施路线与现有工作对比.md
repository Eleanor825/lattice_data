# Lattice 第二方向：做成“垂域数据编译器”

## 1. 这个方向是什么

如果上一条主线是：

> **方向 A：数据价值建模 + 自动数据组合优化**

那么我建议你可以再准备一条更容易起步、也更容易快速沉淀资产的路线：

> **方向 B：Lattice 作为垂域数据编译器（Domain Data Compiler）**

它的目标不是先去解决“最优数据价值公式”这个更难的问题，而是先解决：

- 如何把网页、论文、数据库、手册、百科、论坛、专利、代码文档这些异构来源
- 自动转成统一 schema 下的高质量训练数据
- 并同时产出多个可直接喂模型的视图

例如：

- pretraining corpus
- instruction tuning data
- grounded QA pairs
- structured knowledge records
- tool-use / retrieval traces

一句话说，这条路线是在做：

> **从原始信息到训练数据的自动“编译器”与数据资产平台。**

## 2. 为什么这条路线值得做

### 2.1 它更符合导师“先把数据弄出来”的思路

你导师其实已经讲得很清楚了：

- 先别把 attention 全放在 agent 上
- 先把高价值数据沉淀出来
- 再把操作这些数据的 library 和系统做出来

这个方向正好满足这三点。

### 2.2 它比“直接 formalize 知识密度”更容易落地

“数据价值/知识密度 formalization”非常有研究味，但第一篇 paper 做这个会比较难，因为：

- 定义太抽象
- 很难一开始就做出 convincing metric
- 容易陷入“指标有了，但没人用”的问题

相反，“数据编译器”更容易：

- 很快做出系统原型
- 很快积累真正的数据资产
- 很快有 benchmark 可以验证

### 2.3 它天然能做成长期平台

如果做得好，这个项目不只是论文，还可能变成：

- 某个领域大家默认会来找数据的网站 / Hub
- 某类 foundation model 数据构建的标准 pipeline
- 后续更多研究工作的基础设施

也就是说，它的“平台价值”比单一算法 paper 更高。

## 3. 我建议的具体定位

### 版本 1：材料/科学领域数据编译器

这是我最推荐的起点。

输入源：

- 科学论文
- 教材 / lecture notes
- 维基和百科类页面
- 专利
- 材料数据库和半结构化表格
- 领域网站和 FAQ

输出：

- 高质量 scientific pretraining corpus
- material property QA / instruction data
- entity-relation-property triples
- grounded reasoning traces
- provenance-linked knowledge records

为什么适合：

- 导师在录音里已经举了材料方向的例子
- 这个方向非常缺统一、高质量、持续更新的数据资产
- 既能做“数据仓库”，也能做“训练收益验证”

### 版本 2：技术/工具/代码文档数据编译器

如果你更想贴近 agent / tool-use 场景，也可以做这个版本。

输入源：

- API 文档
- issue / forum / StackOverflow
- 官方手册
- SDK examples
- notebook / tutorial

输出：

- doc-grounded instruction data
- tool-use traces
- retrieval corpus
- troubleshooting QA

这个版本更产品化，但论文上容易被看成“工程整合”。如果想要更强学术说服力，材料/科学版本更好。

## 4. 每一步该怎么做

下面这条路线我尽量写成“能直接开干”的版本。

| 阶段 | 你具体要做什么 | 为什么这一步重要 |
|---|---|---|
| Step 1. 选定单一垂域和任务 | 先只选一个领域，例如 materials；再只选 1-2 个目标任务，例如 continued pretraining + grounded QA | 不先收敛，系统会失控。影响力大的项目通常一开始都很窄，但做得很深 |
| Step 2. 画 source map | 列出该领域最重要的 5-8 类来源：papers、百科、数据库、专利、专业网站、教程、论坛 | 这一步决定数据覆盖面，也决定你后面能否形成“别人会来用”的资产 |
| Step 3. 定义 minimal schema set | 先只定义 4-6 个 schema，如 `Document`、`KnowledgeRecord`、`InstructionTrace`、`StructuredRecord` | 没有 schema，后面所有抽取、过滤、重写都只能写死在脚本里，无法复用 |
| Step 4. 搭 ingestion pipeline | 为不同 source 写统一入口：抓取、下载、解析、OCR/PDF 解析、网页清洗、表格抽取 | 这是整个系统的入口，没有稳定 ingestion，数据资产无法持续增长 |
| Step 5. 做 normalization 和 provenance | 给每条数据记录来源、时间、许可证、版本、引用链接、去重 ID | 这决定系统是否可信、可复现、可公开发布 |
| Step 6. 做 quality filtering | 去重、去 boilerplate、去模板噪声、去错误 OCR、去低信息密度样本 | 原始抓下来不等于能训练；如果没有这个环节，模型收益会很差 |
| Step 7. 做 knowledge extraction / rewriting | 把原始文档转成更高密度的训练视图，例如 QA、triples、属性表、instruction 对 | 这是“数据编译器”和普通清洗管线的分水岭：你不是只清洗，而是在生成更有训练价值的数据形态 |
| Step 8. 做 dataset views | 同一份原始资产导出多种视图：pretrain view、SFT view、retrieval view、knowledge graph view | 同一数据被多次利用，资产价值显著提升，也更容易形成平台壁垒 |
| Step 9. 建 benchmark 和评估 protocol | 定义 held-out perplexity、domain QA、retrieval、知识抽取准确率、训练 token 效率 | 没有统一评估，这个项目永远只能停留在“看起来很有用” |
| Step 10. 做训练验证 | 训练一个小到中等规模 baseline，与通用数据、启发式过滤数据做对比 | 必须证明这个数据编译器带来真实训练收益，否则很难成为强 paper |
| Step 11. 建公开 Hub | 网站 / Hugging Face / 文档站，公开数据卡、pipeline、版本日志 | 如果想做出影响力，这一步几乎是必须的。影响力来自“被使用”，不只来自 paper |
| Step 12. 再加自动化 agent | 在系统稳定之后，再做 agent 来自动调度 ingestion、抽取、检查和发布 | 这样 agent 才是增强层，而不是空壳卖点 |

## 5. 建议的 3 个里程碑

### Milestone 1：先做出一个能跑通的最小编译器

目标：

- 只接 3 类数据源
- 只支持 3 个 schema
- 只输出 2 种训练视图

完成标准：

- 能从 raw source 自动得到训练样本
- 有 provenance
- 有基本去重和过滤

### Milestone 2：证明这份数据确实比“直接抓下来”更值钱

目标：

- 和原始 corpus、简单过滤 corpus 比
- 在小模型上证明 token efficiency 或下游效果更强

完成标准：

- 有 1 张主结果表
- 有 1 张 ablation 表
- 能回答“收益来自哪里”

### Milestone 3：把它做成别人会用的资产

目标：

- 开源一版数据
- 开源 pipeline
- 开网站 / dataset card / docs

完成标准：

- 别人可以下载、复现、更新
- 这个项目不依赖你个人手工维护

## 6. 为什么做这个重要

### 学术上重要

现在很多 foundation model 研究默认“数据已经在那里”。但现实是：

- 真正有价值的数据通常是异构的
- 格式混乱
- 来源分散
- 有大量噪声和重复
- 很难直接变成训练样本

所以“如何把原始世界知识编译成训练数据”本身就是核心问题，不是附属工程。

### 工程上重要

每个组、每个公司都在重复做以下事情：

- 到处找数据
- 自己写爬虫
- 自己做解析和清洗
- 自己做格式对齐

这是一种巨大的重复劳动。如果 Lattice 变成标准化编译器，这部分成本会被大幅压缩。

### 长期上重要

模型架构会快速迭代，但数据资产和数据管线会更持久。真正能沉淀下来的通常是：

- 数据库
- 标准 schema
- 处理工具链
- benchmark
- 社区入口

这也是为什么你导师一直强调“数据才是目的”。

## 7. 这个方向会不会影响力很大

### 短答案

**有潜力很大，但前提是你不能把它做成“又一个数据清洗脚本集合”。**

### 什么时候影响力会很大

如果同时满足下面四点，影响力会很强：

1. **选对领域**  
   这个领域数据分散、门槛高、需求强，例如 materials、biomed、semiconductor、legal。

2. **做成真正资产，而不是 demo**  
   不只是代码仓库，还要有公开数据、版本、文档、网站、benchmark。

3. **证明训练收益**  
   不是“解析得更漂亮”，而是“模型训出来更好”。

4. **持续更新**  
   一次性数据集的影响力有限；可更新的数据平台更容易成为标准入口。

### 什么时候影响力不会太大

如果最后只是：

- 抓了一堆数据
- 做了几步清洗
- 没有 schema
- 没有 benchmark
- 没有训练收益证明
- 没有公开平台

那它更像内部工程项目，而不是高影响力研究。

### 我的判断

如果你把它做成：

> “某一垂域最好的公开数据编译器 + 最有用的数据资产库 + 可证明提升训练效果的 pipeline”

那它的影响力可以非常大。它甚至可能比单篇算法 paper 更耐用，因为别人会持续依赖它。

## 9. 为什么这条方向和上一条方向互补

两条方向不是竞争关系，而是前后关系：

- **方向 B 数据编译器** 先解决“把好数据稳定造出来”
- **方向 A 数据价值建模** 再解决“这些数据怎么最优使用”

更实际的策略是：

1. 先做方向 B，快速积累资产和系统
2. 再在这个资产库上做方向 A 的 mixture / curriculum / valuation

这样论文线也更清楚：

- Paper 1: 垂域数据编译器 / dataset + system
- Paper 2: 数据价值建模 + 自动 mixture optimization

## 10. 我给你的最终建议

如果你现在要和导师讨论“下一步到底先做什么”，我建议你优先提这句话：

> 我们可以先不把重点放在 agent，而是先把 Lattice 做成一个垂域数据编译器：把论文、网页、数据库等异构来源自动转成统一 schema 下的高质量训练数据，并导出 pretraining、QA、instruction、knowledge record 等多种数据视图。这样先把数据资产和工具系统沉淀下来，后续再在这个基础上做数据价值建模和自动 mixture optimization。

这句话会比较稳，因为它既对齐你导师的想法，也给了一个明确、可落地、可扩展的起点。
